RISC-V 32-bit instructions formate

RV32I Instruction Set Architecture Deisgned:

    a) fixed 32-bit instruction keeps decode and fetch simpler i.e. no mixed instruction lengths

    b) a 3-address style register fields (rd, rs1, rs2) need less instructions than 2-address

    c) Encoding and Decoding is relatively uniform with the same bit positioning for rs1, rs2, rd, expect sign-extended immediates 

    1) sufficient to form a compiler target,
        - reliably generate neccessary machine code
    2) can support modern operating system environments

    3) Reduce hardware required in a minimal implementation
        - important to test functionally on the Artix-7

    4) ~40 unique instructions 

Instruction File - Base Instruction Formats (R/I/S/U):

    a) IALIGN = 32, each instruction must start at an address that’s a multiple of 2.

    b) Error detection for invalid branch or jump address, raising misaligned exception

    c) Memory / PC use 8-bit-byte addressing (standard RV32I): one address = one byte (USART map).
       Offsets in I/S/B/J/U types are still counted in bytes, not 16-bit words.

    Immediates are sign-extended (not MIPS-style mixed zero/sign) to keep decode simple.

    --- Byte offsets vs alignment (RISC-V default vs RV-DIS policy) ---

    RISC-V default (Unprivileged ISA):
    - Every PC, branch, jump, load/store uses BYTE addresses (8 bits per address).
    - Branch/jump immediates (B-type, J-type: BEQ, JAL, …) scatter offset bits in the instruction
      and leave imm[0] out of the encoding; decode appends imm[0]=0 (see imm_b / imm_j in
      rtl/common/spu_lite_pkg.sv). That means the offset is always EVEN (multiple of 2 bytes).
    - Purpose: control-flow targets can sit on 2-byte boundaries (for optional RVC compressed insns).
    - This is NOT “16-bit per address” and NOT “multiply offset by 2 at execute” — the decoded value
      is already the full byte offset; other ISAs often store offset/2 and shift left in hardware.

    RV-DIS stricter policy (this project):
    - We only fetch 32-bit instructions (no RVC), so PC and branch/jump/load/store offsets should be
      multiples of 4 (word aligned).
    - After standard decode, imm_align4() in spu_lite_pkg.sv forces imm[1:0]=00 (clear low two bits).
      Valid asm (+4, +8, …) is unchanged; an illegal +2 offset would become 0 (truncate down, not round up).
    - Example: jal r0, 4 → byte offset 4 → insn 0x0040006F; jal r0, 7 bytes is NOT encodable in RV32I
      (odd offset); use +6 or +8. Next 32-bit insn after PC is jal rd, 4 not jal rd, 2.

    Summary table:
    | Layer              | imm[0]   | imm[1:0] | Typical jump +4      |
    |--------------------|----------|----------|----------------------|
    | RISC-V ISA (B/J)   | forced 0 | any even | allowed (+4, +6, …)  |
    | RV-DIS imm_align4  | 0        | forced 00| only +4, +8, …      |

    Branch/jump encoding philosophy (RISC-V “rotation”, no execute-time <<1):
    - Offset bits are permuted in the 32-bit instruction so LSB of offset is zero in the encoding.
    - Decode rebuilds the byte offset (imm_b / imm_j + 1'b0); no extra shifter in odd_lane pc+imm.

Integer Computational Instruction

    a) Integer computational instructions are either encoded as register-immediate operations using the I-type
    format or as register-register operations using the R-type format. 

    b) The destination is register rd for both register-immediate and register-register instructions. 
    
    c) No integer computational instructions cause arithmetic exceptions.

    d) No special overflow-detecting arithmetic instructions, simple branch patterns can detect overflow cheaply, without extra hardware.

        add   t0, t1, t2
        bltu  t0, t1, overflow -> raise software expection

        complier decided whether overflow must be checked, inserting overflow branch instruction

Registers File - 32 addressible register:
    // A conventional size that balances encoding, performance, and hardware simplicity
    a) why more entry (>16) reduce presssure for aggressive compiler optimatizations technieuqes(loops, caching) that uses many temporary register

    b) why less entry (<64) a few register dominate in use, get much of the benefits without needing to use all register equally often
    
    1) Register x0 is hardwired with zero

    2) General Purpose registers x1-x31 (RV32I Unprivileged ISA, "Register State")

        ABI / convention (not extra hardware in a minimal core):
        - x1 (ra)  — usual return-address register for JAL / JALR
        - x2 (sp)  — stack pointer
        - x5 (t0)  — may be used as an alternate link register in some code sequences

        RV-DIS: JAL/JALR write the link (pc+4) to whatever GPR rd encodes — no special x1/x5
        hardware; compilers usually pick x1 (ra), but any rd except x0 is valid.

    3) Program counter — 32-bit byte address (Unprivileged ISA: PC is not a GPR)

        Instruction memory: 8-bit byte map (one address = one byte). Each 32-bit instruction
        occupies four byte addresses; fetch assembles ILEN=32 into the instruction register.
        RV-DIS uses PC aligned to 4 bytes for 32-bit instructions (no RVC in base bring-up). 

    4) Scalar GPR read ports — 2 (RISC-V) × 2 (dual-issue) = 4

        a) 2 read ports per instruction (RV32I)
        - 3-address encoding: rs1, rs2 fields in every format (I-type uses imm for 2nd operand)
        - One insn needs 0–2 GPR reads (e.g. ADD/BEQ/SW use both; ADDI/LW use rs1 only)

        b) × 2 for dual-issue → 4 read ports total
        - Even lane: dedicated rs1 + rs2 ports (ALU / OP-IMM)
        - Odd lane:  dedicated rs1 + rs2 ports (load/store / branch)
        - Both lanes can read GPR in the same ID cycle (e.g. even ADD + odd SW)

    5) Scalar GPR write ports — 2 (dual-issue)

        a) One write port per lane, same writeback cycle
        - Even lane writes rd from ALU result (when rd != x0)
        - Odd lane writes rd from load data, JAL/JALR link, or LUI/AUIPC

        b) Unified 32×32 GPR; merge rules
        - Same rd from both lanes on one WB: younger insn wins (compare latched PC)
        - Issue unit stalls RAW hazards; forwarding between even and odd lanes

    6) Vector register file (SIMD) — separate 8×128-bit VR; extra read/write ports, not in scalar 4+2 above










    

128-bit SIMD Extension (Cell/NEON-style, SPU-Lite):

    a) 128-bit vector width; 8 vector registers v0-v7 (separate from scalar x0-x31)

    b) Lane modes: 16x8-bit, 8x16-bit, 4x32-bit packed integer ops

    c) Even lane: VADD, VSUB, VAND, VOR, VXOR (128-bit SIMD ALU)

    d) Odd lane: VLD128, VST128 (16-byte aligned vector load/store)

    e) Dual-issue example: vector ALU (even) + vector/scalar load (odd) in same cycle
