RISC-V 32-bit instructions formate

RV32I Instruction Set Architecture Deisgned:

    a) fixed 32-bit instruction keeps decode and fetch simpler i.e. no mixed instruction lengths

    b) a 3-address style register fields (rd, rs1, rs2) need less instructions than 2-address

    c) Encoding and Decoding is relatively uniform with the same bit positioning for rs1, rs2, rd, expect sign-extended immediates 

    1) sufficient to form a compiler target,
    - reliably generate neccessary machine code

    2) can support modern operating system environments

    3) Reduce hardware required in a minimal implementation
    - important to test functionally on the Artix-7

    4) ~40 unique instructions 

Program counter — 32-bit byte address (Unprivileged ISA: PC is not a GPR)

    Instruction memory: 8-bit byte map (one address = one byte). Each 32-bit instruction
    occupies four byte addresses; fetch assembles ILEN=32 into the instruction register.
    RV-DIS uses PC aligned to 4 bytes for 32-bit instructions (no RVC in base bring-up). 

Instruction File - Base Instruction Formats (R/I/S/U):

    1) IALIGN = 32, each instruction must start at an address that’s a multiple of 2.

    2) Error detection for invalid branch or jump address, raising misaligned exception

    3) Memory / PC use 8-bit-byte addressing (standard RV32I): one address = one byte (USART map).
       Offsets in I/S/B/J/U types are still counted in bytes, not 16-bit words.

    4) Immediates are sign-extended (not MIPS-style mixed zero/sign) to keep decode simple.

        --- Byte offsets vs alignment (RISC-V default vs RV-DIS policy) ---

        a) RISC-V default (Unprivileged ISA):
        - Every PC, branch, jump, load/store uses BYTE addresses (8 bits per address).
        - Branch/jump immediates (B-type, J-type: BEQ, JAL, …) scatter offset bits in the instruction
        and leave imm[0] out of the encoding; decode appends imm[0]=0 (see imm_b / imm_j in
        rtl/common/rv_dis_pkg.sv / decode_pkg.sv). That means the offset is always EVEN (multiple of 2 bytes).
        - Purpose: control-flow targets can sit on 2-byte boundaries (for optional RVC compressed insns).
        - This is NOT “16-bit per address” and NOT “multiply offset by 2 at execute” — the decoded value
        is already the full byte offset; other ISAs often store offset/2 and shift left in hardware.

        b) RV-DIS decode policy (this project):
        - We only fetch 32-bit instructions (no RVC); PC should stay 4-byte aligned in normal code.
        - imm_align4() in decode_pkg.sv forces imm[1:0]=00 on branch/jump offsets (B, J, JALR).
        Load/store/OP-IMM use full sign-extended 32-bit byte offsets (no align4).
        - B/J still have imm[0]=0 in the ISA encoding; decode appends that bit (see imm_b / imm_j).

            Summary table:
            | Layer              | imm[0]   | imm[1:0] | Typical jump +4      |
            |--------------------|----------|----------|----------------------|
            | RISC-V ISA (B/J)   | forced 0 | any even | allowed (+4, +6, …)  |
            | RV-DIS imm_align4  | 0        | forced 00| B/J/JALR only        |

        Branch/jump encoding philosophy (RISC-V “rotation”, no execute-time <<1):
        - Offset bits are permuted in the 32-bit instruction so LSB of offset is zero in the encoding.
        - Decode rebuilds the byte offset (imm_b / imm_j + 1'b0); no extra shifter in odd_lane pc+imm.
            
Instruction - Integer Computational 

    1) Integer computational instructions are either encoded as register-immediate operations using the I-type
    format or as register-register operations using the R-type format. 

    2) The destination is register rd for both register-immediate and register-register instructions. 
    
    3) No integer computational instructions cause arithmetic exceptions.

    4) No special overflow-detecting arithmetic instructions, simple branch patterns can detect overflow cheaply, without extra hardware.

        add   t0, t1, t2
        bltu  t0, t1, overflow -> raise software expection

    complier decided whether overflow must be checked, inserting overflow branch instruction

Dual Decoder :

    1) Parallel decode — two insns per cycle
    - I0 = older (lower PC), I1 = younger (PC+4): opcode decode, control signals, lane_sel (even / odd)

    2) Lane differatation is determined(lane_sel)
    - PC age track the order of execution not the lane(even/odd)

Registers File - 32 addressible register:
    // A conventional size that balances encoding, performance, and hardware simplicity
    0.1) why more entry (>16) reduce presssure for aggressive compiler optimatizations technieuqes(loops, caching) that uses many temporary register

    0.2) why less entry (<64) a few register dominate in use, get much of the benefits without needing to use all register equally often
    
    1) Register x0 is hardwired with zero

    2) General Purpose registers x1-x31 (RV32I Unprivileged ISA, "Register State")

        ABI / convention (not extra hardware in a minimal core):
        - x1 (ra)  — usual return-address register for JAL / JALR
        - x2 (sp)  — stack pointer
        - x5 (t0)  — may be used as an alternate link register in some code sequences

        RV-DIS: JAL/JALR can write the link (pc+4) to whatever GPR rd encodes — no special x1/x5
        hardware; compilers usually pick x1 (ra), but any rd except x0 is valid.

    3) Scalar GPR ports (RISC-V) x (dual-issue)

        a) 2 read ports per instruction (RV32I)
        - 3-address encoding: rs1, rs2 fields in every format (I-type uses imm, rs2 = 0x0)
        - Instruction may needs 0–2 GPR reads (e.g. ADD/BEQ/SW use both; ADDI/LW use rs1 only)

        b) multiply by 2 for dual-issue as lanes → 4 read ports total
        - Even lane: dedicated rs1 + rs2 ports (ALU / OP-IMM)
        - Odd lane:  dedicated rs1 + rs2 ports (load/store / branch)
        - Both lanes can read GPR in the same ID cycle regardless of even or odd
         (e.g. even ADD + odd SW, even ADD + even SUB, odd BNE + odd BGE)

    5) Scalar GPR write ports — 2 (dual-issue)
        a) 2 write ports for dual-issues as lanes
        - Both lanes can write GPR in the same WB cycle depending on PC-age(ignore older instruction)
        - Lanes are differatiated at WB, dedicated even/odd WB with no adventage

    6) structure Hazards, for immediate values
        a) immediate instruction is decoded with r2 as zero register 

            addi x2, x1, 0xaa
            => even_rs1_addr = x1, even_rs2_addr = x0
            * immediate value(0xaa) is not visible to the register file
            * desintation register(x2) is not visible to the register file
            * rd = rs1/rs2 is structually sounded with read/write port
    
        b) single instruction, read from the same register 

            add x2, x1, x1
            => even_rs1_addr = x1, even_rs2_addr = x1
            * in this case even_rs1_data = even_rs2_data

            Likewise: 
            beq x2, x2, label 
            => odd_rs1_addr = x2, odd_rs2_addr = x2
            * always brench to label, since odd_rs1_data = odd_rs2_data

        - rs1 = rs2 is structually sounded with 2 read ports

        c) dual instruction, read from the same register
        - structurally sounded with with 2 set of 2 read ports
        
        b) dual instruction write back odd/even_wdata lane write enable
        - structurally sounded with 2 set of write port(independent ctrl signals)


    7) data hazards, register read and write back
        a) RAR (Read After Read) — dual instructions both read the same register

            add x3, x3, x1 | bne x3, x2
            => even_rs1_addr = x3, even_rs2_addr = x1
            => odd_rs1_addr = x3, odd_rs2_addr = x2

            * fundemental loop exit check routine
            * in this case even_rs1_data = odd_rs1_data, structually true, but logically fault

            Likewise: 
            addi x3, x3, 0x0a | sub x3, x3, x1 
            => even_rs1_addr = x3, even_rs2_addr = x1
            => odd_rs1_addr = x3, odd_rs2_addr = x2
            * fundemental loop count down starting from 0x0a
            * lane are not differatiated yet, this is structurally llegal, but logically fault

        b) Register RAW / WAR at GPR writeback (I0 = older, I1 = younger)
        - register file: combinational read from regs[] only; writes commit to regs[] on negedge clk
        - no WB bypass on GPR read ports; no falling-edge forward into EX operands
        - RAW (Read After Write): in-flight rd not in regs[] — EX uses forward_unit MEM-stage
          mux (mem0/mem1) only; otherwise stall until producer visible at MEM or committed in regs[]
        - WAR (Write After Read): I0 reads old value from regs[] before I1 WB negedge commit
        - WAW: both write same rd — both issue; register_file merges at WB (younger wpc wins, see c)

        c) WAW (Write After Write) — dual instructions both write the same register

            addi x2, x1, 0xaa | lw x2, 0x0(x2) 
            => even_pc = 0x0, even_rd = x2, even_data = 0xaa
            => odd_pc = 0x4, odd_rd = x2, odd_data = 0xbb, let 0x0(x2) = 0xbb
            * retrive from memory by immediate pointer address

        - Dispatch does not suppress either insn on same rd (JAL/JALR PC and other side
          effects must still execute; skipping I0 is not equivalent to running then merging)
        - Both I0 and I1 reach their lanes; register_file commits one value per rd at WB
          negedge — younger insn wins via wpc when both write the same rd same cycle
        - if Software expects 0xaa as an observable value, complier must space these instructions
          outside the pair.

Dispatch (during ID/EX) — lane checks and hazards

    Dispatch runs at ID_EX (before operands enter EX). It decides whether the fetch pair
    I0 + I1 issue to the execution lanes. Operand forwarding at EX is forward_unit only
    (MEM stage mem0/mem1); no WB bypass and no falling-edge forward from GPR commit.

    1) Lane map:
    - Even lane: OP, OP-IMM (ALU)
    - Odd lane:  LOAD, STORE, BRANCH, JAL, JALR, LUI, AUIPC

    3) structure hazard - Same instruction type need the same lanes even\even or odd\odd

        a) Clean pair:
            addi x1,x5,0x2c | lw x2,0x00(x5)
            => I0 even, I1 odd → lane check passes

        - no conflict with dual lanes, even and odd lanes 
        - no stall

        b) Lane conflict: I0 and I1 need the same even lane 

            add x1,x2,x3 | sub x4,x5,x6
            => both I0, I1 need LANE_EVEN 

            likewise:

            lw x1, 0x00(x2) | beq x2, x3, label

            or 

            lw x1, 0x00(x3) | lw x2, 0x00(x3)
            => both I0, I1 need LANE_ODD 

        - duplicated resources, 2 sets of ports to execute the same type of instruction in Parallel
        - two even lanes and two odd lanes, 4 sets of output port in total with individual enable

            I0 is even type         => ev0_[port]
            I1 is also even type    => ev1_[port]    

            I0 is odd type          => od0_[port]
            I1 is also odd type     => od1_[port]

        - no stall

    4) data hazard - register focus (I0 = older, I1 = younger)

        a) RAR (Read After Read) no semantic conflict, dual GPR read ports
        - dual read in register file(7a) and dual read from memory cache

        b) WAR (Write After Read), I0 must see pre-write value (anti-dependency)
        - resolve by pipeline staging

        c) WAW (Write After Write), same memory word (same rd handled at WB — §4.c)
        - resolve at write back, kill older (I0) i.e dont't wback the older instruction

        d) RAW (Read After Write): I1 reads rs1/rs2 that equals an in-flight rd
        - reader needs the writer's new value; 

        d.1) 1 stall required - skipping memory access stage 

            Even then Even — intra-dependent
            addi x5, x4, x3 | xor x7, x6, x5
            => I0 rd = x5, I1 rs2 = x5

            Even then odd — inter-dependent
            addi x5,x5,0x01 | brne x6, x5, label
            => I0 rd = x5, I1 rs2 = x5

        - hazard unit / scoreboard at ID→EX: detect I1 rs1/rs2 == I0 rd (I0 reg_write, rd != x0)
        - same cycle: issue I0 only; hold I1 in ID (stall IF/ID as needed)
        - next cycle: forward_unit supplies I0's result from MEM stage (mem0/mem1); then issue I1
        - no WB bypass; no falling-edge forward; no same-bundle EX forward (see §6)

        d.2) 2 stall required - read/write to memory 

            Odd then even - inter-dependent
            lw x2,0x00(x5) | addi x1,x2,0x2c
            => I0 rd = x2, I1 rs1 = x2

            Odd then odd - intra-dependent
            lw x2,0x00(x5) | lw x5,0x00(x2)
            => I0 rd = x2, I1 rs1 = x2

        - load data is not available until s4_memory; MEM-stage forward cannot help same cycle
        - I1 must be buffered and pipeline stalled until x2 is ready
        - stall required for all cases of load from memory is the first instruction

        e) Memory RAW / WAR — same eff. addr, load/store ordering (2 stall, buffered at dispatch)

            WAR — I0 load, I1 store same word:
            lw x7, 0(x5) | sw x6, 0(x5)

            RAW — I0 store, I1 load same word:
            sw x6, 0(x5) | lw x7, 0(x5)

        - Detect at dispatch (eff. addr match: rs1+imm, same 32-bit word)
        - In-order pipeline: issue I0 only; buffer I1 in dispatch hold (same mechanism as d.1/d.2)
        - 2 stall cycles until I0 memory op completes (store visible / load data returned)
        - Then replay I1 — memory_cache receives ordered requests only; no RAW/WAR logic in cache
        - Same-bundle memory WAW: kill I0 write at issue (see c, Cache §6) — not stalled, suppressed

    5) control hazard - 
            
        Odd then even - no stall branch instruction
            brne x5,x6,label | addi x6 

    6) Scoreboard (issue gate at ID→EX) 

        Purpose:
        - Central in-order issue gate for GPR RAW hazards
        - Same-bundle RAW (d.1): single-issue I0, push suppressed insn into insn_buffer (4 deep)
        - In-flight RAW: operand ready = !busy[rs] OR MEM forward hit (mem0/mem1 only)
        - Load-use (d.2): rd stays busy until WB retire; replay when load data on MEM stage
        - stall_id to IF/ID; stall_mem ORs memory_cache miss back-pressure only

        busy[31:0]:
        - Set when an instruction with reg_write (rd != x0) issues into EX
        - Cleared when that rd retires at WB (ex_mem_wb merged wb0/wb1)

        insn_buffer — 4-entry FIFO (dispatch_funct/insn_buffer.sv, in id_ex_dispatch):
        - Holds individual suppressed instructions in program order (head = oldest)
        - Push on set_i1_hold: full ID bundle + producer_rd (I0 rd) + bundle PC tags
        - Pop on issue_i1_from_hold when head operands ready (MEM fwd_hit on producer_rd)
        - push_ok when count <= DEPTH-2 (>=2 free slots); else global stall_id on all front-end stages
        - While push_ok=0: stall fetch/decode; replay buffer head when ready to drain (stall_id stays 1)
        - Shared queue for scoreboard GPR RAW (§4.d) and future dispatch memory holds (§4.e)
        - i1_hold_active = !empty (legacy name: any buffered insn waiting)
        - suppress_bundle_raw_q: after head replays, ignore same PC pair until IF/ID advances

        Module ports (scoreboard.sv):
        - ID bundle I0/I1: valid, reg_write, rd, rs1/rs2, rs1_use/rs2_use
        - insn_buffer head: rs1/rs2 use+addr, producer_valid/rd; buf_empty, buf_push_ok
        - Forward-ready: mem0/mem1 reg_write+rd only (fwd_hit); no wb0/wb1 forward path
        - stall_mem external; suppress_bundle_raw; hold_replay_block (push cycle)
        - Outputs: issue_i0, issue_i1, stall_id, set_i1_hold, issue_i1_from_hold, bundle_raw

        Hazard ownership (id_ex_dispatch vs scoreboard):

        scoreboard.sv — GPR register RAW only (§4.d):
        - d.1 same-bundle (1 stall): single-issue I0, hold I1, replay when MEM fwd_hit
        - d.2 load-use (2 stall): busy[rd] until WB retire; hold I1 until load data ready

        id_ex_dispatch — checked at dispatch, not in scoreboard:
        - Memory RAW/WAR (§4.e): same eff. addr (rs1+imm); single-issue I0; buffer I1;
          2-stall in-order replay when I0 mem op completes; memory_cache sees ordered ports only
        - GPR WAW: no dispatch check; register_file wpc merge at WB (§4.c)
        - Memory WAW (§4.c, Cache §6): kill I0 store at issue (od0 mem_en=0)
          (reg_write=0 / mem_en=0) — redundant write never executes

        Other:
        - EX operand mux: forward_unit MEM stage only (mem0/mem1); no WB bypass; no falling-edge forward
        - memory_cache: L1 miss back-pressure (stall_mem) only

    7) Stall methods

        Default: global front-end stall (b). Same-bundle RAW (§4.d.1) also uses partial issue (a).

        a) Partial issue — one fetch pair only
            - Same-bundle RAW: issue I0; push I1 to insn_buffer (4-deep FIFO); replay head on MEM fwd_hit
            - Cross-bundle lane run-ahead not supported

            Example — even|even RAW (§4.d.1):
                addi x5, x4, x3 | xor x7, x6, x5
                c0: issue ev0 (addi); push xor to buffer; stall_id=1
                c1: mem0 forwards x5 → replay head on ev1; stall_id=0

            Example — even|odd RAW:
                addi x5, x5, 1 | bne x6, x5, label
                c0: issue ev0; buffer branch (od1); stall_id=1
                c1: mem0 forwards x5 → replay head on od1

        b) Global stall_id — IF/ID bubble until dispatch can proceed
            - insn_buffer head not ready (MEM fwd / load data)
            - push_ok=0 (<2 free buffer slots): stall fetch; replay head to drain; stall_id stays 1
            - load-use (§4.d.2), memory RAW/WAR (§4.e), cache miss (stall_mem)
            - stall_id && !issue_i0 && !issue_i1 → hold EX registers

            Example — load-use (§4.d.2):
                lw x2, 0(x5) | addi x1, x2, 0x2c
                c0: issue od0 (lw); push addi to buffer; stall_id=1
                c1–N: stall until x2 at MEM/WB → replay addi on ev1

            Example — buffer backpressure (push_ok=0, depth 4):
                buffer holds 3+ insns → <2 free slots
                stall_id=1 (no new fetch); replay head each cycle when ready until count≤2

            Example — memory RAW (§4.e):
                sw x6, 0(x5) | lw x7, 0(x5)
                c0: issue od0 (sw); buffer lw; stall_id=1
                c1–N: stall until store completes → replay lw on od1

        c) Back-end per-slot freeze (optional, inactive)
            - ex_mem stall_od0/od1, ex_mem_wb stall_i0/i1 — one slot only; miss uses stall_mem today

        | Situation                      | Method                          |
        |--------------------------------|---------------------------------|
        | Same-bundle RAW (d.1)          | (a) partial issue + buffer push |
        | Buffer <2 free slots           | (b) global stall_id             |
        | Load-use / mem order / L1 miss | (b) global stall_id             |
        | Cross-bundle independent lanes | not supported                   |

        Signals: stall_id, stall_mem; issue_i0/i1, set_i1_hold, issue_i1_from_hold; stall_od0/od1, stall_i0/i1


        
Forwarding (EX operand mux — MEM stage only)

    Policy — not used in RV-DIS:
        - no WB bypass on GPR read ports (no same-cycle wen→read short-circuit for EX)
        - no falling-edge forward (operands never sourced from negedge GPR commit in the same cycle)

    forward_unit muxes EX rs operands from:
        a) ID/EX GPR operand (combinational read of regs[] — stale if producer still in flight)
        b) MEM stage mem0/mem1: ex_mem_wb EX bank (even) or odd mem mux
        - youngest PC wins when mem0 and mem1 both match the same rs
        - after WB negedge commit, the next ID read sees the new value in regs[] (no forward needed)
        - does not resolve same-bundle I0→I1 RAW; scoreboard stalls that case (d.1)
        - same-bundle GPR WAW: both issue; younger wpc wins at register_file WB
        - same-bundle memory WAW: I0 store suppressed at dispatch (od0 off) — §4.c, Cache §6



Execution Flow and Design:

    1) Dual Issues, odd/even lanes
        a) even lanes executes scaler operation(R-type, I-type)
        b) odd lanes executes branch and memory access operation(B-type, S-type, U-type)

    2) Duplicate execution resources (port-based EX)
        a) Avoid stall-based lane arbitration by duplicating excution units on the same lane
        b) Dispatch both I0, I1 assign to even/odd lane to the (2) copy of execution unit 
        c) Removes I0, I1 same type structural hazards at the cost of more hardware


Cache Memory Design:

    1) Byte addressing — 8-bits per address(same as intruction cache) as standard RV32I data memory map   
    - Address N and N+1 are adjacent bytes
    - Effective address: mem_addr = rs1 + imm (byte offset from decode, same unit as PC)
    - Load/store offsets: sign-ext only (no imm_align4); use 4-byte-aligned addresses in software(complier)

        lw x5 0(x10) - correct align
        lw x5 1(x10) - miss align
        lw x5 2(x10) - miss align
        lw x5 3(x10) - miss align
        lw x5 4(x10) - correct allign

    2) LW / SW (32-bit) — mem_besel selects which bytes out of a 32-bit word will be write/read at the mem_addr

    - write and read in little-endian

    write/read data  |  in memory
    0xddccbbaa =>       0x00000 : aa  
                        0x00001 : bb 
                        0x00010 : cc 
                        0x00011 : dd

    3) RAR (Read After Read) — dual-issue read same memory word
    - structurally sound with dual read ports

        lw x5, 0(x6) | lw x7, 0(x6)
        => same eff. addr; I0 read may be suppressed if only I1's result is architecturally needed

    - disable redundant I0 mem port at dispatch when appropriate

    4) WAR (Write After Read) — I0 read, I1 write, same memory word (anti-dependency)

        lw x7, 0(x5) | sw x6, 0(x5)
        => I0 load must read memory before I1 store overwrites the word

    - dispatch: detect same eff. addr; single-issue I0; buffer I1 (2-stall in-order path)
    - replay I1 after I0 load completes; memory_cache assumes ordered requests only

    5) RAW (Read After Write) — I0 write, I1 read, same memory word (true dependency)

        sw x6, 0(x5) | lw x7, 0(x5)
        => I1 load depends on the value I0 store writes to the word
    
    - dispatch: detect same eff. addr; single-issue I0; buffer I1 (2-stall in-order path)
    - replay I1 after I0 store completes; memory_cache assumes ordered requests only

    6) WAW (Write After Write) — dual-issue write same memory word

        sw x6, 0(x5) | sw x7, 0(x5)
        => same eff. addr; I1 (younger) is the architecturally visible store

    - checked at dispatch (id_ex_dispatch): kill I0 store at issue (od0 mem_en=0)
    - only I1 store reaches memory_cache; no dual write to the same word in one cycle

    7) L1 cache miss
    - stall to fetch the data in lower cache levels  

Even lane at EX/WB (skips MEM)

    1) even-lane ALU result latches in ex_mem_wb EX bank at posedge (no memory access)
    2) odd-lane reg_write uses ex_mem_wb MEM bank (ex_mem → load/JAL/LUI mux)
    3) ex_mem_wb merges even or odd per slot to GPR write ports


Memory hierarchy — Basys 3 (XC7A35T / Artix-7):

    On-chip block RAM (FPGA BRAM, not SPI flash):
    - Device total: ~1,800 Kbit ≈ 225 KiB (50 × 36 Kbit RAMB36 tiles)
    - RV-DIS on-chip cache SRAM budget must stay within this pool (plus separate BRAM for

128-bit SIMD Extension (Cell/NEON-style, SPU-Lite):

    a) 128-bit vector width; 8 vector registers v0-v7 (separate from scalar x0-x31)

    b) Lane modes: 16x8-bit, 8x16-bit, 4x32-bit packed integer ops

    c) Even lane: VADD, VSUB, VAND, VOR, VXOR (128-bit SIMD ALU)

    d) Odd lane: VLD128, VST128 (16-byte aligned vector load/store)

    e) Dual-issue example: vector ALU (even) + vector/scalar load (odd) in same cycle
